## 快速入门：5 分钟写一个 Skill

### 示例：中文技术文章转公众号

```bash
# 创建 Skill 目录
mkdir -p ~/.openclaw/skills/wechat-article
```

创建 `SKILL.md`：

```markdown
---
name: wechat-article
description: 将技术文章转换为微信公众号风格的文章
---

# 技术文章转公众号

## 使用场景
当用户说"转公众号风格"、"改成公众号文章"时使用。

## 执行步骤
1. 阅读原文，提取核心观点
2. 改写标题：使用吸引眼球的标题格式
3. 添加开头：用一个引人入胜的问题或故事开头
4. 改写正文：
   - 段落短小（3-5行）
   - 多用类比和比喻
   - 加入 emoji 适当点缀
   - 关键句子加粗
5. 添加结尾：总结 + 互动引导
6. 输出 Markdown 格式

## 风格要求
- 口语化，像朋友聊天
- 避免学术腔调
- 每段话不超过 100 字
- 标题格式：数字 + 关键词 + 情绪词
  例如："3 个让你代码速度翻倍的技巧，第2个太实用了！"
```

### 测试

在 OpenClaw 中说：

```
使用「技术文章转公众号」Skill，帮我把这篇文章改成公众号风格：
[粘贴你的技术文章]
```

## 更多示例

### 示例2：每日站会报告生成器

```markdown
---
name: daily-standup
description: 根据 Git 提交记录自动生成每日站会报告
---

# 每日站会报告

## 使用场景
当用户说"生成站会报告"、"今天的standup"时使用。

## 执行步骤
1. 运行 `git log --since="yesterday" --oneline` 获取提交记录
2. 运行 `git diff --stat HEAD~5` 查看最近修改
3. 按以下格式生成报告：

## 报告模板

### 昨天完成了什么
- [根据 git log 总结]

### 今天计划做什么
- [根据未完成的分支和 TODO 推断]

### 有什么阻碍
- [如果有未解决的 merge conflict 等，列出来]
```

### 示例3：代码审查助手

```markdown
---
name: code-review
description: 对代码进行专业审查，给出改进建议
---

# Code Review

## 使用场景
当用户说"review 代码"、"帮我看看这段代码"时使用。

## 检查清单
1. **安全性**：SQL注入、XSS、敏感信息泄露
2. **性能**：N+1 查询、内存泄漏、不必要的循环
3. **可读性**：命名规范、注释质量、函数长度
4. **最佳实践**：错误处理、边界条件、类型安全
5. **测试**：测试覆盖率、边界测试

## 输出格式
- 🔴 严重问题（必须修复）
- 🟡 建议改进（推荐修复）
- 🟢 优点（值得表扬的地方）
```
