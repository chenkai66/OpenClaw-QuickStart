# 第6章·第6节：Agent 设计哲学与最佳实践

> 用好 AI Agent 的关键不在于模型多强，而在于你如何构建它的知识体系和运行环境。

---

## 一、为什么 Agent 的核心竞争力是知识文件

很多人以为 Agent 的能力完全取决于底层模型。其实模型只是"智商"，而 workspace 里积累的知识文件才是"经验"。两个人用同样的模型，体验可能天差地别——差距在于各自积累了什么。

这个道理和现实世界一样：两个智力相当的工程师，工作三年和工作一天的人，产出完全不同。区别在于积累的经验和方法论。

### Agent 的知识飞轮

整个知识积累机制可以用一个循环来概括：

```
每次对话前 → 加载 workspace 中的知识文件到 prompt
    ↓
对话过程中 → Agent 执行任务、发现新信息、犯错、被纠正
    ↓
对话结束时 → 把新学到的东西写回知识文件
    ↓
下次对话 → 加载更新后的文件 → 行为更精准
    ↓
循环往复，能力不断增长
```

这个简单循环构成了一个强大的飞轮：用得越多，积累越多，效果越好。

### 每个知识文件的价值

| 文件 | 作用 | 类比 |
|------|------|------|
| SOUL.md | Agent 的人格与底线 | 一个人的价值观 |
| USER.md | 对你的画像 | 同事对你的了解 |
| AGENTS.md | 做事规矩和踩坑记录 | 工作手册 + 经验总结 |
| SKILL.md | 特定领域操作规范 | 专业技能证书 |
| MEMORY.md | 提炼后的长期记忆 | 笔记精华 |
| memory/*.md | 每日原始记录 | 工作日志 |
| TOOLS.md | 环境备忘 | 工位说明书 |

代码决定了 Agent 能做什么，知识文件决定了它做得多好。

## 二、模型选择：框架再好也需要强底座

Agent 框架本身不生产智能，它只是让模型发挥得更好的放大器。框架再好，底层模型不行，上限就在那里。

### 选模型的核心原则

- **关键任务用最好的模型**：方案设计、复杂推理、反思总结这类任务，模型能力直接决定质量
- **日常任务用性价比模型**：定时推送、格式化输出、简单查询这类任务，中等模型足够
- **心跳健康检查用本地模型**：纯粹的存活检测，不需要强推理，本地小模型即可

### 80/20 成本法则

80% 的任务用中等模型控制成本，20% 的关键任务用最强模型保质量。不要追求所有任务都用最贵的模型，也不要为了省钱在关键环节用弱模型。

---

## 三、专精优于通才——善用多 Agent

一个什么都做的 Agent，就像一个什么岗位都兼的员工——什么都会一点，什么都不精通。

### 为什么要拆分 Agent

每个 Agent 有独立的工作空间、独立的记忆、独立的对话历史。这意味着一个专做某个领域的 Agent，它积累的所有经验都与该领域相关，不会被无关对话稀释。

经验积累是垂直的、深度的，而不是水平的、稀薄的。

### 拆分原则

- **按领域拆分**，不按任务类型拆分
- **每个 Agent 只负责一个职能**，让知识积累保持纯粹
- **不要过度拆分**——Agent 数量增加，协调成本急剧上升（N 个 Agent 之间有 N(N-1)/2 对交互关系）

---

## 四、培训你的 Agent——知识沉淀的方法论

装好 Agent 就指望它跟用了三个月一样好使，就像新员工第一天就要求他跟老员工一样高效。Agent 需要培训。

### 主动引导，不要被动等待

不要等 Agent 自己慢慢摸索。在你已经有成熟方法论的领域，直接把流程写成 SKILL.md 让 Agent 执行。

```
你已有的经验  →  写成 SKILL.md  →  Agent 每次按规范执行  →  效果立竿见影
```

### 纠错是最有效的培训

在 Agent 犯错时明确纠正它，它会把教训记录下来。几次纠正之后，同类错误的复发率会显著下降。

关键是要让经验固化——临时的对话纠正会随着会话结束而消失，只有写入知识文件的经验才是永久的。

### 定期审查知识文件

Agent 自己写的内容不一定都准确。定期检查 AGENTS.md、USER.md、MEMORY.md 里有没有过时或偏差的内容，及时修正。

---

## 五、上下文管理——Agent 系统的核心工程问题

上下文窗口是 Agent 最宝贵的资源，管理不好一切都白搭。

### 信息分层原则

不是所有信息都该放在最显眼的位置。要按优先级分层：

1. **身份层（SOUL.md）**：最核心的约束和人格，始终加载，保持精简
2. **规范层（AGENTS.md）**：操作规范和协作协议
3. **技能层（SKILL.md）**：按需加载，不用的技能不占上下文
4. **记忆层（MEMORY.md）**：提炼后的关键经验
5. **临时层（对话历史）**：自动管理，超限压缩

### 上下文的"热数据"与"冷数据"

和操作系统的内存管理一样：热数据放缓存（SOUL.md），冷数据放磁盘（memory 归档），关键数据常驻内存（MEMORY.md）。

### 知识文件不能无限膨胀

框架对文件大小有预算限制（单文件通常限制在 20KB 以内）。这意味着 Agent 需要学会"提炼"，把最重要的经验浓缩在有限空间里。这也是为什么 MEMORY.md（精华）和 memory/*.md（原始日志）要分开的原因。

---

## 六、记忆的分层治理

Agent 的记忆如果只有一层，要么全部丢失，要么全部混乱。好的记忆系统应该像人脑一样分层。

### 推荐的分层模型

| 层级 | 存储 | 时间尺度 | 管理方式 |
|------|------|---------|---------|
| 身份层 | SOUL.md | 永久 | 人工确认修改 |
| 长期记忆 | MEMORY.md | 长期 | Agent 自主维护 |
| 中期记忆 | memory/*.md | 中期 | 框架自动提取 |
| 短期记忆 | 当前对话 | 临时 | 自动压缩 |
| 持久知识 | Skills + 归档 | 永久 | 共享 / 归档 |

### 自我进化的关键：反思机制

Agent 犯了错，不能只在当场改正，还要有机制让它把经验写回知识文件：

```
犯错/被纠正 → 即时记录 → 定期反思（评估重要性）→ 重要经验提升到长期记忆 → 下次自动避免
```

这个循环是 Agent 从"chatbot"升级为真正"Agent"的分水岭。

### 一个关键安全原则

Agent 可以修改自己的记忆和工作规范，但不应该修改自己的"灵魂"（SOUL.md）。改人格的权限只能在人手里——否则 Agent 可能会把自己的约束放松，行为变得不可控。

---

## 七、规则设计的铁律

### 用最强措辞，面向最弱模型

在多模型环境下，你无法确定每次推理用的是哪个模型。所以所有关键规则都要按最弱模型的理解能力来写：

- `"建议不要编造数据"` → 弱模型几乎无视
- `"MUST: 禁止编造数据"` → 遵守率显著提升
- `"MUST + 最高优先级 + 不可协商"` → 即使弱模型也能遵守

**规则设计的核心：显式大于隐式，硬规则大于软建议，可预测大于可解释。**

### SOUL.md 要精而不杂

SOUL.md 不是"行数越少越好"，而是信息密度和优先级排列要合理。如果把操作规范、临时指令、协作协议全堆在 SOUL.md 里，关键约束容易被稀释。

- SOUL.md 只放**身份定义 + 硬约束 + 决策框架**
- 操作规范放 AGENTS.md
- 领域知识通过 Skills 按需加载

## 八、多 Agent 协作的协议设计

把多个 Agent 放在一起不等于它们能协作——就像把几个工程师拉到一个没有流程的群聊里一样混乱。

### 协作需要协议，不是群聊

两个 Agent 在群聊里互相回复会导致无限循环（A 回复 B → B 回复 A → 无限循环）。解决方案不是"告诉它们少说话"，而是设计有状态机的通信协议。

### 推荐的三态通信模型

```
发起方发出请求 (request)
    ↓
接收方确认并回复结论 (confirmed)
    ↓
发起方发出终态确认 (final)
    ↓
双方进入静默，不再互相回复
```

每条消息带唯一标识符，同一轮交互只允许一个标识符。终态发出后禁止续话。

### 异步优于同步

Agent 之间的关键数据传递不要完全依赖实时消息，应该同时落地到共享文件。消息通道可能超时、重复，但文件是持久的、可追溯的。

## 九、生产环境的运行哲学

### 持续运行的系统必然退化

这不是"可能"，是"一定"。配置堆积、记忆膨胀、会话增长、磁盘撑满——这些会确定性地发生。

对策不是"一次配好永远不管"，而是建立**反退化机制**：

- 会话自动压缩和重置
- 记忆定期归档和清理
- 配置变更自动备份
- 健康状态定期巡检

### 90% 的时间花在工程问题上

运行 Agent 系统更像是做运维（SRE），而不是做 AI 研究。会话膨胀、消息风暴、配置被改坏——这些问题的解法在分布式系统和运维经验里，不在 AI 论文里。

### "智能"有时是灾难

在需要精确、可预测输出的场景下，AI 的"智能优化"反而有害。比如数据报告被"智能压缩"丢失了关键表格，配置文件被"智能修复"改出了新问题。

**可预测 > 可解释 > 不可控的"智能"。**

## 十、核心实操清单

按优先级排序：

| 优先级 | 实践 | 说明 |
|--------|------|------|
| 1 | 选对模型 | 框架是放大器，模型是基础，基础不行上限就在那里 |
| 2 | 会话管理第一天就配好 | 空闲超时、自动重置、磁盘上限——不配等于定时炸弹 |
| 3 | SOUL.md 保持精简 | 只放核心约束，非核心内容迁移到 AGENTS.md 和 Skills |
| 4 | 主动写 SKILL.md | 不要等 Agent 自己摸索，成熟的流程直接固化成技能 |
| 5 | 善用多 Agent | 不同领域配不同 Agent，让经验积累保持垂直 |
| 6 | 定期审查知识文件 | Agent 自己写的内容可能过时或不准确 |
| 7 | 规则用最强措辞 | 面向最弱模型设计，`MUST` 而不是 `SHOULD` |
| 8 | 备份你的 workspace | 这是你最有价值的数字资产，用 Git 管理，定期推送 |
| 9 | 建立反思机制 | 让 Agent 定期从错误中学习，把经验固化 |
| 10 | 监控不能只靠 Agent | 独立于框架的健康检查才是可靠的 |
