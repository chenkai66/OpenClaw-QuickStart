# 第9章·第3节：生产环境踩坑指南

> 装好 Agent 只是起点。这些经验来自真实的长期运行实践，帮你避开最常见的深水区。

---

## 一、从 1 个 Agent 开始，逐个添加

这是最重要的一条建议。

一次性启动多个 Agent 会导致：级联响应（Agent 之间互相触发回复风暴）、多个 SOUL.md 同时未调试（行为不可预测）、会话资源竞争（同时膨胀导致系统崩溃）、问题叠加无法定位（5 个 Agent 同时出问题时，你甚至分不清是哪个导致的）。

**正确做法**：每加一个 Agent，先单独运行半天到一天，确认行为正常后再加下一个。每次只改一个变量——如果同时加 Agent + 改配置 + 换模型，出了问题根本无法定位。

---

## 二、会话管理是生死线

会话（Session）管理是最容易忽略但最致命的问题。它的表现症状极具迷惑性——Agent 开始反复压缩内容、输出重复的话，或者回复跟之前说过的几乎一模一样。遇到这些表现，十有八九是会话上下文过长了。

### 必须从第一天就配好的参数

| 参数 | 建议值 | 不配的后果 |
|------|--------|----------|
| 空闲超时重置 | 30 分钟 | 凌晨的会话到早上还活着，跨天数据残留 |
| 自动清理时限 | 7 天 | 会话文件堆积到几个 GB |
| 磁盘上限 | 100MB | 磁盘被撑满 |
| 每日定时重置 | 凌晨 5 点 | 昨天的数据污染今天的分析 |

### 会话膨胀的隐蔽场景

- **多个定时任务共享会话**：一天几十个定时任务在同一个会话中执行，上下文越滚越大。高频任务应该配置为每次新建会话
- **心跳检查累积**：健康检查每小时执行一次，每次的输出都追加到同一个会话。几天后会话膨胀到几万 token，反而导致 Agent 卡死
- **压缩不是实时的**：框架的会话压缩只在特定触发条件下执行（如空闲超时），两次触发之间会话会持续膨胀

---

## 三、Agent 会改坏自己的配置

当 Agent 有文件写入权限时，它可能会"修复"自己的配置文件——把有效的工具名改成无效的，把约束条件放松，甚至修改自己的"人格"。

最诡异的是：没有人改过配置，是 Agent 自己改的。它有修改的能力，但没有判断后果的能力。

**防线**：
1. 关键配置文件每天自动备份
2. 检测到异常修改时自动恢复
3. 考虑收紧执行权限，只允许白名单命令
4. 绝对不允许 Agent 修改 SOUL.md（灵魂文件）

---

## 四、消息风暴——"积极汇报"变成阻塞攻击

让 Agent 去做一个调研任务，它不断向主频道发送进展更新——"调研中"、"找到结果了"、"补充细节"、"确认归档"、"完成通知"——连发大量消息。

问题不只是刷屏。在会话串行处理模型下，每条消息都要排队处理。Agent 间的消息堆积在队列里，用户发的消息被排在最后——用户只看到"Agent 不理我了"。

**解决思路**：
- 非紧急任务的进展更新应该写入文件，不走消息通道
- 同一轮交互只允许固定数量的消息往返
- 终态消息发出后，双方必须进入静默

---

## 五、双向 @ 导致无限循环

当两个 Agent 被配置为"只在被 @ 时才回复"，而它们互相 @，就会产生经典的 ACK 风暴：

```
A @ B → B 回复并 @ A → A 收到并 @ B → B 回复并 @ A → ……
```

每条回复可能都言之有物，但对话永远不会停。这不是因为 Agent"太客套"，而是因为**缺少终态定义**。

**解决方案**：设计有状态机的通信协议。明确定义"请求 → 确认 → 终结"三个阶段，终结后双方必须静默。不要试图靠"告诉 Agent 少说话"来解决——文档级的建议在长上下文中很容易被忽视。

---

## 六、投递成功不等于归档成功

用户在聊天频道收到了 Agent 的报告，但知识库里没有记录。Agent 说"一切正常"，但实际产出为零。

核心原因：**很多人把"消息发出去了"等同于"任务完成了"**。实际上：

- 消息投递成功 ≠ 内容已归档
- 没有报错 ≠ 有有效产出
- Agent 说"正常" ≠ 真的正常
- 状态字段写了"已完成" ≠ 产物真的存在

**解决思路**：对关键任务设置硬性检查——产出文件是否存在、内容是否非空、关键字段是否完整。"建议检查"不够，要"必须检查"。

---

## 七、"智能压缩"吃掉你的数据

框架在消息过长时会自动做"智能压缩"——用 AI 来概括和缩减内容。听起来很智能，但在数据密集的场景下是灾难。

一份包含完整数据表格、详细指标的报告，被"智能压缩"后，表格消失了，数字被概括了，关键数据丢失了。框架认为"帮你优化了"，但你需要的恰恰是那些被删掉的细节。

**解决思路**：
- 数据密集型输出绕过框架的压缩管线
- 了解框架的文本长度阈值，必要时手动控制输出长度
- 关键数据同时落盘到文件，不完全依赖消息通道

---

## 八、SOUL.md 信息过载导致规则失效

当 SOUL.md 堆满了各种操作规范、临时指令、协作协议时，Agent 会开始"选择性遵守"规则。它不是变笨了，而是关键信息被噪声淹没了。

管生活的 Agent 越界去做投资分析。处理数据的 Agent 忽略了数据验证规则。

**解决思路**：
- SOUL.md 只放核心身份和硬约束，保持精简
- 操作规范放到 AGENTS.md
- 领域知识通过 Skills 按需加载
- 核心约束放在文件最前面（前面的内容权重更高）

---

## 九、免费模型的隐性成本

为了省钱尝试免费模型或第三方代理，结果往往是：

- 免费模型在工具调用等关键场景下完全不可靠——参数丢失、格式错误
- 第三方代理服务不稳定，随时可能中断
- 免费额度用完后定时任务半夜中断，无人知晓
- 弱模型对规则的遵循率远低于强模型，精心设计的约束形同虚设

**核心原则**：免费的往往是最贵的。在关键任务上用可靠的付费服务，在日常任务上用性价比方案，两者分开管理。

---

## 十、自我改进系统的递归陷阱

让 Agent 从对话中学习、自动记录经验，是一个很好的想法。但如果没有硬性的递归深度限制，"自我改进"行为本身会触发更多"自我改进"——形成无限循环。

Agent 识别到一个"需要记录的经验" → 调用工具记录 → 记录行为被识别为"新的状态变化" → 再次触发记录 → 无限循环。会话文件暴增，资源持续消耗直到系统崩溃。

**防线**：
- 每条用户消息最多触发 1 条记录
- 记录行为本身不能触发新的记录
- 设置工具调用次数上限
- 区分"用户纠正"和"正常讨论"——不是所有对话都需要学习

---

## 十一、进程活着不等于服务正常

系统级的进程守护（如 macOS 的 launchd）只管进程是否存活。但以下故障，进程守护一个都发现不了：

- 会话膨胀导致推理超时（进程活着，但处理不了请求）
- 配置被改坏导致启动循环（反复崩溃重启，但每次都失败）
- 子进程泄漏导致内存耗尽（主进程正常，子进程堆积）
- 浏览器实例不关闭导致资源占满

**解决思路**：需要在进程守护之上建立应用级的健康检查——不只看"进程在不在"，还要看"能不能正常响应"、"资源用量是否正常"、"关键任务是否在产出"。

---

## 十二、子任务完成检测的缺失

委派子 Agent 执行任务后，你可能发现框架**没有主动通知你任务什么时候完成**。没有回调，没有事件通知，什么都没有。

这意味着：
- 你只能轮询或猜测子任务是否完成
- 超时不等于失败——可能只是响应慢，任务还在跑
- 僵尸会话累积，最终打满并发限制

**解决思路**：
- 超时后不要立即重发（可能导致重复执行）
- 子任务的完成判定应该基于产物是否存在，而不只是状态字段
- 需要独立的轮询和清理机制

## 诊断速查表

当你遇到问题时，按这个顺序排查：

| 症状 | 最可能的原因 | 检查方向 |
|------|------------|---------|
| Agent 反复说同样的话 | 会话上下文过长 | 检查会话文件大小 |
| Agent 不回复 | 会话被其他消息阻塞 | 检查是否有排队消息 |
| Agent 越界做不该做的事 | SOUL.md 信息过载 | 检查文件行数和关键约束位置 |
| Agent 突然"忘了"偏好 | 会话压缩丢失上下文 | 关键偏好是否写入 MEMORY.md |
| 配置莫名其妙变了 | Agent 自己修改了配置 | 对比配置文件备份 |
| 定时任务没有产出 | 任务执行了但输出为空 | 检查产出文件是否存在 |
| 两个 Agent 无限对话 | 缺少终态协议 | 检查通信规则 |
| 系统越来越卡 | 子进程/浏览器实例泄漏 | 检查进程列表和内存占用 |
| 模型降级后行为异常 | 弱模型不遵守规则 | 检查规则措辞强度 |
| 子任务状态不明 | 缺少完成通知机制 | 手动检查子任务产出 |

---

## 核心原则总结

1. **渐进式扩展**：从 1 个 Agent 开始，逐个验证后再添加
2. **会话即生命**：从第一天就配好超时、清理、磁盘限制
3. **信任但验证**：Agent 说"完成了"不等于真完成，要看产物
4. **分层治理**：不同重要级别的信息放在不同的文件层级
5. **协议优于提示**：多 Agent 协作靠状态机协议，不靠"少说话"
6. **面向最弱设计**：规则用最强措辞，因为你不知道哪次会用到弱模型
7. **独立监控**：守护系统不能依赖被守护的对象
8. **反退化意识**：持续运行的系统一定会退化，主动建立清理机制
9. **备份一切**：workspace 是最有价值的资产，配置要自动备份
10. **免费最贵**：关键任务用可靠方案，省错地方代价更大
